方案 A: GraphQL + Apollo

灵活查询，精确获取所需字段，减少 over-fetching

+ 精确查询 + 强类型 Schema + 实时订阅 - 学习曲线陡
详细分析
适合数据关系复杂、客户端需求多样的场景。Apollo Client 提供缓存、乐观更新、错误处理开箱即用。社区生态成熟，DevTools 完善。缺点是服务端需要额外的 Schema 设计成本，简单 CRUD 场景收益不明显。

方案 B: REST + React Query

传统 REST API 搭配客户端缓存层

+ 团队熟悉 + 缓存自动管理 - Over-fetching - 多端适配难
详细分析
最稳妥的选择。团队无需额外学习成本，React Query 的 stale-while-revalidate 策略解决了大部分缓存问题。但当多个端（Web、Mobile、Mini Program）对同一 API 有不同字段需求时，维护成本快速上升。

方案 C: tRPC

端到端类型安全，零 Schema 定义

+ 端到端类型安全 + 零 API Schema - 仅限 TS 全栈 - 生态较新
详细分析
TypeScript 全栈项目的理想选择。前后端共享类型定义，IDE 自动补全覆盖 API 调用。但要求前后端都用 TypeScript，且对非 TS 客户端（移动端原生）不友好。适合内部工具、SaaS 后台等单一技术栈项目。

什么是 GraphQL？
GraphQL 是一种 API 查询语言，由 Facebook 于 2012 年内部开发，2015 年开源。它的核心思想是：让客户端精确描述需要什么数据。 
query {
  user(id: "u_01") {
    name
    orders(first: 3) {
      items { productName price }
      total
    }
  }
}
与 REST 的区别：REST 每个端点返回固定结构，GraphQL 一个端点返回客户端指定的结构。
Apollo Client 缓存机制
Apollo Client 使用 normalized cache（归一化缓存）：每个对象按 __typename:id 作为唯一键存储。当多个 Query 返回同一个对象时，它们共享同一份缓存，一处更新全局生效。 
// 缓存结构示意
{
  "User:u_01": { "name": "Alice", "email": "alice@example.com" },
  "Order:o_99": { "total": 299, "user": { "__ref": "User:u_01" } }
}
这意味着：修改 User:u_01 的 name，所有引用该用户的 Order 也会自动更新。
什么是 N+1 查询问题？
当 Resolver 嵌套时，外层查 N 条记录，内层每条再查一次，产生 N+1 次数据库查询。 
// 问题示例
users() → SELECT * FROM users (1次)
  user.orders() → SELECT * FROM orders WHERE user_id = ? (N次)
    order.items() → SELECT * FROM items WHERE order_id = ? (N*M次)
解决方案：使用 DataLoader 将同一 tick 内的多个查询合并为一次批量查询。效果：320ms → 45ms。
乐观更新 (Optimistic UI)
用户操作后立即更新界面，不等待服务端响应。如果服务端失败则回滚。关键在于给每次操作一个临时 ID。 
const [addTodo] = useMutation(ADD_TODO, {
  optimisticResponse: {
    addTodo: { id: 'temp-' + Date.now(), text: input, done: false }
  },
  update(cache, { data }) {
    // 成功: 用真实 ID 替换临时 ID
    // 失败: 自动回滚到操作前状态
  }
});
用户体感：点击 = 立即生效，延迟 = 0ms。
Fragment 共享模式
将组件需要的字段定义为 Fragment，避免 props drilling 和字段遗漏。 
fragment UserAvatar on User {
  id name avatarUrl
}

fragment OrderSummary on Order {
  id total createdAt
  user { ...UserAvatar }
}

# 任何组件都可以声明自己依赖哪个 Fragment
好处：组件自描述需要什么数据，修改字段不会影响其他组件。
查询复杂度分析
GraphQL 允许客户端构造深度嵌套查询，可能造成服务端资源耗尽。解决方案：设置查询复杂度上限。 
// 每个字段分配 cost
const costMap = {
  users: 1,        // 基础列表
  user: 1,         // 单条查询
  orders: 5,       // 关联查询 cost 更高
  items: 10,       // 深层嵌套 cost 最高
};
// 限制: 单次查询 cost ≤ 1000
超限查询直接拒绝，返回 400 + 明确的错误信息。
持久化查询 (Persisted Queries)
客户端发送查询的 hash 而非完整 query string，服务端从缓存中查找对应的查询。好处：减少网络传输体积，防止恶意查询注入。 
// 客户端: 只发送 hash
{ "hash": "a1b2c3d4e5f6...", "variables": { "id": "u_01" } }

// 服务端: hash → 缓存中找到完整 query → 执行
APQ (Automatic Persisted Queries) 可以自动完成这个过程，无需手动注册。
